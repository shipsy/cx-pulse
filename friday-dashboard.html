Where F.R.I.D.A.Y Excels vs Struggles
Patterns observed across all sampled tickets. The primary bottleneck for accuracy is tooling access, not reasoning quality — when F.R.I.D.A.Y can reach the data, it almost always gets the answer right.

Strengths (100% accuracy in these areas)

Feature requests & info queries — always classifies correctly with code-level evidence
API payload validation errors — traces code path from input to output, identifies missing fields
Data mapping issues — pincode, address, config mapping problems well-handled
Investigation depth — structured multi-layer trails far more detailed than typical human RCA (30-40 KB logs)

Weaknesses (accuracy gaps)

Backend services without ES logs (OBB-V2, Bull queues) — no Elasticsearch access = speculation
Client-side / "not our issue" cases — over-investigates instead of recognizing simpler truth
DTDC org data access — HTTP 500 on getMagicLinkData blocks investigation for highest-volume account
Duplicate ticket detection — doesn't check if ticket duplicates an existing one
Operational context — misses business context a human would immediately recognize

How Every Metric Is Calculated
Full transparency on data sources, formulas, and methodology. Every number in this dashboard is derived from two primary sources: the Slack channel #friday-tkt-runs (C0AU1LXBEPR) and DevRev ticket data.

Total Tickets (2,298)

All DevRev support tickets created between Apr 30 and May 31, 2026.

Source: DevRev list_tickets API with created_date filter

Agent Runs (523)

Count of distinct ticket-level Shipra bot posts in #friday-tkt-runs. Each ticket produces one "run" (the first message with *Ticket:* pattern).

Source: Slack channel message scan, deduplicated by ticket ID

Coverage (22.8%)

Percentage of total tickets that F.R.I.D.A.Y processed.

Formula: 523 / 2,298 = 22.76%

RCA + FR Draft (68)

Runs where BOTH "*Agent Response:*" AND "*Customer-Facing Draft:*" messages appear for the same ticket.

Pattern: Two Shipra messages per ticket — rca.md + customer-response.md

RCA Only (318)

Runs where "*Agent Response:*" appears but NO "*Customer-Facing Draft:*" follows for that ticket.

Pattern: One Shipra message — rca.md only

Skipped (133)

Runs where "*Skipped:* Workspace is not mapped to a Shipsy org_id." appears.

Pattern: Shipra message with "Skipped" keyword

Failed (4)

Runs where "*Investigation failed:*" appears. Reasons: process aborted by user (3), API fetch error (1).

Pattern: Shipra message with "Investigation failed" keyword

RCA Accuracy (84.6%)

20-ticket sample compared F.R.I.D.A.Y's RCA against human-provided resolution (ctype__root_cause_and_resolution_details field + dev_user comments).

Formula: (6 match + 5 partial) / 13 comparable = 84.6%

Timing (Median 4.2 min, Avg 5.3 min)

Measured from Slack timestamps: delta between DevRev bot notification (B059SMU1611) and Shipra output (U0AUP74EXLL) for same ticket ID. 37 RCA samples, 13 skip samples collected from May 19-30.

Formula: Shipra_message_ts - DevRev_message_ts = seconds

Recommendations for Improvement
Prioritized actions to increase F.R.I.D.A.Y's coverage, accuracy, and value. Ordered by impact-to-effort ratio.
HIGH

Map unmapped workspaces

133 runs (25.4%) are skipped because customer workspaces aren't mapped to Shipsy org_ids. Mapping these would immediately increase RCA coverage from 73.8% to near 99%.

Impact: +25.4% coverage
Effort: Medium
HIGH

Fix DTDC data access (getMagicLinkData)

DTDC is the highest-volume account but F.R.I.D.A.Y gets HTTP 500 errors when fetching consignment data for org "1". Fixing this would significantly improve RCA quality for Tier 1.

Impact: Tier 1 accuracy
Effort: Medium
MEDIUM

Add duplicate ticket detection

F.R.I.D.A.Y doesn't check if a ticket duplicates an existing one, wasting investigation effort. Adding a similarity check before deep-diving would save compute and improve signal quality.

Impact: Fewer wasted runs
Effort: Low
MEDIUM

Add "not our issue" classifier

F.R.I.D.A.Y over-investigates when the actual answer is "problem is on the customer/partner side." Training on resolution patterns like "resolved at client's end" would reduce false hypotheses.

Impact: Accuracy +5-10%
Effort: Medium
MEDIUM

Expand ES log coverage

Backend services like OBB-V2 and Bull queues don't log to the Elasticsearch indices F.R.I.D.A.Y can search. This is the primary reason for RCA mismatches.

Impact: Eliminate mismatch root cause
Effort: High (infra)
LOW

Increase FR draft generation rate

Currently only 17.6% of RCAs get a customer-facing draft. Tuning the confidence threshold or providing more customer communication templates could increase this significantly.

Impact: More customer value
Effort: Medium
